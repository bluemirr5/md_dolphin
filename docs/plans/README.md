## 이 폴더는 무엇인가

마스터 플랜이 사이클 5 종료 시점까지 1140줄로 비대해져, sub-agent마다 필요한 단락이 다름에도 매번 전체 파일을 부분 read해야 했다. 본 인덱스는 마스터 플랜을 **8개 파일로 분할**하고, sub-agent별 read 가이드와 사이클 종료 시 갱신 체크리스트, cross-reference 코드 위치 표를 제공한다.

**분할 원칙**:

- **자주 read되는 단락**(사이클 표·변경 이력)은 **단독 파일**로
- **거의 동결된 단락**(비전·기술 결정·디자인 토큰)은 **한 파일에 묶어** 1회 read로 끝내기
- **갱신 빈도가 다른 단락**(변경 이력 vs 결정)은 **분리** — 사이클마다 변경되는 곳이 좁아짐
- **파일당 200~400줄 권장** — Read 도구 25k 토큰 제한 안에 단일 파일 read 가능

## sub-agent별 read 가이드

각 sub-agent가 반드시 read해야 하는 파일과, 갱신해야 하는 파일을 분리한다.

### spec-writer

**Read (사이클 N 스펙 작성 시)**:

1. `03-cycles.md` — 사이클 표(6.1)에서 N행 + 의무 기재 항목(6.7)
2. `05-changelog.md` — 사이클 N에 흡수된 부채 메모(0.1 표 끝부분)
3. (필요 시) `01-decisions.md` — cross-reference 코드(P2-5, P2-8 등) 인용 시
4. (필요 시) `04-deps-impact.md` — 사이클이 새 의존성을 도입할 때

**Write**: 없음. spec 파일은 `docs/specs/cycle-NN-<name>.md`에 작성.

**금지**: `00-vision.md`, `02-design-tokens.md`, `06-validation.md`는 평소 read 불필요. 사이클 4 디자인 토큰 spec 작성 시에만 `02`를 read.

### architect

**Read (스펙 리뷰 시)**:

1. `01-decisions.md` 전체 — 결정과 정합 검증
2. `03-cycles.md` — 해당 사이클 행 + 의존 사이클
3. `05-changelog.md` — 변경 이력의 P{N} 라운드 기록(중복 제기 방지)
4. (필요 시) `04-deps-impact.md` — 미지·외부 의존 영향
5. (필요 시) `00-vision.md` — Non-Goals 위배 검증

**Write**: 리뷰 결과는 spec 파일의 변경 이력에 P{N}-{i} 형식으로 기록. 마스터 플랜 갱신은 doc-writer 또는 planner의 책임.

### developer

**Read**: 거의 마스터 플랜 참조 안 함. spec 파일만 읽고 구현.

- 예외: spec에 cross-reference 코드(예: P2-7)가 인용되어 있고 의미가 모호할 때만 인덱스의 cross-ref 표에서 위치 찾아 1회 read.

### code-reviewer

**Read**:

1. spec 파일 전체
2. (필요 시) `01-decisions.md` — 구현이 결정과 어긋나는지 점검
3. (필요 시) `02-design-tokens.md` — UI 사이클(4·6·7) 리뷰 시

### doc-writer (drift 점검 + 사이클 종료 갱신)

**Read**:

1. `03-cycles.md` — 사이클 표(6.1) 현재 상태
2. `05-changelog.md` — 변경 이력 표(0.1)
3. `04-deps-impact.md` — 미지 표(7), 운영 계획(9)
4. spec 파일 전체 + 실제 구현 파일 일부

**Write (사이클 종료 시 갱신 체크리스트는 다음 절 참조)**:

- 주로 `05-changelog.md` 0.1 표에 행 추가
- 사이클 표(6.1)의 해당 사이클 행에 부채 메모 1줄 추가
- 미지 해소 시 `04-deps-impact.md` 7장 표 행 갱신 (해소 시점·완료)

## 사이클 종료 시 갱신 체크리스트 (doc-writer 의무)

사이클 N 종료 시 doc-writer가 수행할 작업. **누락 0 보장**을 위해 순서대로 진행.

### 1. `05-changelog.md` 갱신 (필수)

- [ ] **0.1 변경 이력 표**에 행 1개 추가 — 형식: `\| YYYY-MM-DD \| doc-writer \| 사이클 N 완료 마킹 \| <핵심 결과 요약 1줄> \|`
- [ ] **0.1 표 끝부분에 부채 메모** — 사이클 N+1으로 끼워넣을 항목이 발견됐다면 한 줄로 기록 (사이클 5→6 부채 메모 형식 참고)

### 2. `03-cycles.md` 갱신 (필수)

- [ ] **사이클 표(6.1)**의 사이클 N 행 "비고" 또는 산출물 칸에 부채/완료 표시 추가 (필요 시)
- [ ] **사이클 표(6.1)**의 사이클 N+1 행에 새 부채 항목이 끼어들었다면 산출물 칸에 1줄 추가

### 3. `04-deps-impact.md` 갱신 (해당 시)

- [ ] **7장 미지 표** — 사이클 N에서 해소된 미지가 있다면 "해소 시점" 칸에 완료 표시. 새로 발견된 미지는 행 추가
- [ ] **8장 외부 의존성** — 사이클 N에서 새 라이브러리가 추가됐다면 행 추가
- [ ] **9장 운영 계획** — 사이클 11a/11b 종료 시에만 갱신

### 4. `06-validation.md` 갱신 (선택)

- [ ] **11장 가정** — 사이클에서 검증된 가정이 어긋났다면 갱신
- [ ] **12장 다음 액션** — 사이클 N+1 액션으로 갱신

### 5. `01-decisions.md` / `02-design-tokens.md` (드물게)

- [ ] 결정·토큰이 변경될 때만 — 변경 시 **반드시** `05-changelog.md` 0.1 표에 P{N}-{i} 형식으로 라운드 기록

### 6. 마지막

- [ ] `git status`로 변경 파일 확인 → spec 파일과 마스터 플랜 분할 파일이 모두 포함되었는지 확인
- [ ] 본 README.md의 "최종 갱신" 메모 갱신 (선택)

## 파일별 절(섹션) 번호 — grep 가이드

cross-reference 코드를 찾기 어려울 때 절 번호로 grep 가능. 절 번호는 분할 후에도 원본과 동일하게 유지된다.

```bash
# 예: 4.4.2 로컬 자산 정책 찾기
grep -n "^#### 4.4.2" docs/plans/01-decisions.md

# 예: 0.1 변경 이력 표 찾기
grep -n "^### 0.1" docs/plans/05-changelog.md

# 예: 사이클 표 찾기
grep -n "^### 6.1" docs/plans/03-cycles.md
```
